# 注册数量扩展、无限注册模式与域名成功率统计设计

## 背景

当前注册页的批量注册能力仍有三个明显限制：

1. 注册数量只支持 `1-100`
2. 无法表达“持续跑，直到满足停止条件”的无限注册场景
3. 批量任务结束后只能看到总成功数和失败数，无法按邮箱域名分析成功率

用户本轮希望把普通批量注册范围扩展到 `1-500`，同时新增一个独立的“无限注册”模式，并在任务结束后按域名统计成功率和失败率，用于快速判断哪些邮箱域名更稳定。

## 目标

- 将普通批量注册数量范围扩展到 `1-500`
- 在注册模式中新增“无限注册”
- 让无限注册真正按运行状态持续派发任务，而不是用固定大数量伪装
- 支持“连续失败超过 10 次自动停止”
- 在无限注册运行中显示清晰的实时状态
- 在批量类任务结束后输出按域名汇总的成功率 / 失败率统计并排序
- 尽量复用现有注册接口、状态接口和前端结构，避免无关重构

## 非目标

- 不改 Outlook 专用批量注册入口的交互模式
- 不把单次注册改造成可长期驻留的守护任务
- 不在本轮引入复杂的任务队列中间件或外部调度器
- 不新增跨任务历史报表页
- 不为域名统计增加单独的数据库报表表

## 用户确认的关键规则

- 普通批量注册数量范围：`1-500`
- 新增独立注册模式：`无限注册`
- 无限注册不使用数量输入框，后端语义为 `count=0`
- 无限注册停止条件：
  - 手动取消
  - 连续失败超过 10 次自动停止
- 连续失败计数规则：只要中间有 1 次成功就立刻清零
- 无限注册进度展示方式：保留进度区，但总数展示为 `∞ / 不限`
- 任务结束后增加按域名统计成功率和失败率并排序的功能

## 总体方案

本轮按两个层面改造：

1. **注册模式与批量执行模型增强**
   - 普通批量：固定数量，范围扩展到 `1-500`
   - 无限注册：新增专用运行器，动态创建并派发任务

2. **结束态统计增强**
   - 任务完成后输出域名维度统计
   - 统计结果直接通过现有批量状态返回给前端展示

整体原则是：

- 前端通过显式模式表达用户意图
- 后端通过 `count=0` 统一承载无限注册语义
- 统计以“实际执行完成的任务”为准，不依赖猜测

## 前端交互设计

### 1. 注册模式

注册模式从当前的：

- 单次注册
- 批量注册

扩展为：

- 单次注册
- 批量注册
- 无限注册

行为如下：

#### 单次注册

- 保持现有行为不变
- 仍调用单次注册接口

#### 批量注册

- 显示数量输入框
- 数量范围改为 `1-500`
- 保留并发模式、并发数、间隔时间配置

#### 无限注册

- 隐藏数量输入框
- 保留并发模式、并发数、间隔时间配置
- 提交时走批量注册接口，但传 `count=0`

### 2. 数量输入与文案

普通批量模式下：

- 标签改为：`注册数量 (1-500)`
- input 改为：`min=1`、`max=500`

无限注册模式下：

- 不展示数量输入框
- 控制台启动日志改为：`正在启动无限注册任务...`

### 3. 批量进度展示

#### 普通批量

保持现有语义：

- 进度：`completed / total`
- 百分比：按固定总数计算
- 剩余数：`total - completed`

#### 无限注册

批量进度区保留，但切换为无限语义：

- 进度文本：`已完成 / ∞`
- 总数文案：展示为 `不限`
- 百分比文案：显示 `运行中` 或 `不限`
- 进度条：改成无限滚动动画，不再使用固定百分比宽度
- 额外展示：`连续失败 x / 10`

这样用户可以同时看到：

- 当前仍在运行
- 已完成多少个任务
- 是否接近自动停机阈值

### 4. 域名统计展示

批量类任务结束后，在批量进度区下方展示域名统计表。

列建议为：

- 排名
- 域名
- 总数
- 成功
- 失败
- 成功率
- 失败率

展示规则：

- 仅在任务结束后显示
- 数据较多时区域固定高度并开启纵向滚动
- 控制台日志追加一条汇总提示，例如：
  - `[统计] 域名成功率统计已生成，共 N 个域名`

## 接口与请求语义设计

### 1. 请求入口

接口保持尽量兼容：

- 单次注册：继续使用 `POST /registration/start`
- 批量注册与无限注册：统一使用 `POST /registration/batch`

### 2. count 语义

`BatchRegistrationRequest.count` 新语义如下：

- `0`：无限注册
- `1-500`：普通批量注册

参数校验更新为：

- 合法范围：`0-500`
- 错误提示：`注册数量必须在 0-500 之间，0 表示无限注册`

### 3. 状态返回扩展

为了避免前端通过 `total=0` 猜测模式，批量状态返回新增：

- `is_unlimited: bool`
- `consecutive_failures: int`
- `max_consecutive_failures: int`
- `stop_reason: completed | manual_cancel | too_many_consecutive_failures | null`
- `domain_stats: []`（仅结束时返回结果）

普通批量仍继续返回：

- `total`
- `completed`
- `success`
- `failed`
- `finished`

这样旧逻辑可以平滑升级，新前端也能稳定区分普通批量和无限注册。

## 后端执行模型设计

### 1. 普通批量注册

普通批量保持“固定数量”模型，但数量上限从 `100` 调整为 `500`。

其执行特点不变：

- 启动前按数量预创建任务
- 并行模式和流水线模式沿用现有实现
- 状态统计继续以固定总数为基础

### 2. 无限注册新增专用运行器

无限注册不再复用“预创建固定数量任务”的模型，而是新增专用运行器，例如：

- `run_unlimited_batch_registration(...)`

该运行器职责：

- 初始化无限任务批次状态
- 按并发模式持续创建并提交注册任务
- 监听任务完成结果
- 根据完成结果维护连续失败计数
- 到达停止条件后停止继续派发新任务
- 等待已在途任务自然完成
- 生成最终汇总与域名统计

### 3. 为什么不能直接用大数量模拟无限

不采用“先塞一个很大的 count”方案，原因是：

- 无法准确表达“无限直到条件满足”为业务语义
- 前端进度和剩余数会失真
- 手动取消和自动停止的边界会更混乱
- 连续失败阈值在预创建任务模型下更难保证可控

因此本轮采用真正的动态派发模型。

## 无限注册运行规则

### 1. 停止条件

无限注册只有两类停止条件：

1. **手动取消**
2. **连续失败超过 10 次自动停止**

精确定义为：

- 当连续失败计数达到第 `11` 次时，停止继续派发新任务

### 2. 连续失败计数规则

用户确认的规则是：

- 只要中间出现 1 次成功，连续失败计数立刻清零

考虑到无限注册支持并发，本轮按**任务完成顺序**统计连续失败，而不是按任务创建顺序。

原因是：

- 并发场景下完成顺序才代表真实运行反馈流
- 可以避免“后创建任务先完成”时计数失真
- 与用户实时看到的成功/失败日志顺序一致

### 3. 超阈值后的处理方式

当连续失败达到第 11 次时：

- 立即停止继续创建和派发新任务
- 已经在运行中的任务允许自然完成
- 待在途任务全部收尾后，将批量任务标记为结束

不建议直接强杀已运行任务，因为那会带来：

- 日志不完整
- 邮箱与会话状态不一致
- 更复杂的资源清理问题

### 4. 并发与派发行为

无限注册仍支持现有两种并发模式：

- `pipeline`
- `parallel`

但其内部实现改为“动态供给任务”，而不是固定任务列表：

#### parallel

- 维持一个最大并发上限
- 某个任务完成后，只要未达到停止条件，就补派下一个新任务

#### pipeline

- 维持最大并发上限
- 任务派发之间继续遵守 interval 规则
- 若达到停止条件，则停止继续补派

本质上，两者都变成“运行中补位”，区别只是补位节奏不同。

## 任务结果与域名统计数据设计

### 1. 统计范围

域名统计适用于所有批量类任务：

- 普通批量注册
- 无限注册
- Outlook 批量注册

统计触发时机为任务结束后，包括：

- 正常完成
- 手动取消后的自然收尾完成
- 无限注册因连续失败超过阈值而自动结束

### 2. 纳入统计的任务

仅统计已经实际执行完成的任务：

- `completed`
- `failed`

不统计：

- 尚未启动的任务
- 中途被取消且未实际执行完成的任务
- 纯占位任务

### 3. 域名来源

为了让失败任务也能参与统计，需要明确保存“任务实际使用的邮箱地址”。

推荐在 `registration_tasks` 上增加一个明确字段，例如：

- `email_address` 或 `generated_email`

写入规则：

- 邮箱创建成功后立即写入
- 后续无论任务成功还是失败，该字段都保留

这样可以避免：

- 依赖 `result` JSON 结构解析邮箱
- 成功任务和失败任务统计口径不一致

### 4. 无邮箱失败的归类

若任务在创建邮箱之前就失败，没有实际邮箱地址，则在统计时统一归类为：

- `未获取邮箱`

这样可以保证：

- 失败任务不会在统计中消失
- 汇总总数和实际完成数一致

### 5. 统计项

每个域名生成一条聚合记录：

- `domain`
- `total`
- `success`
- `failed`
- `success_rate`
- `failure_rate`

计算规则：

- `total = success + failed`
- `success_rate = success / total`
- `failure_rate = failed / total`

### 6. 排序规则

默认排序规则：

1. 成功率降序
2. 成功率相同时，总数降序
3. 仍相同时，域名字母升序
4. `未获取邮箱` 永远排最后

这样可以优先把“样本量更大且成功率更高”的域名放在前面。

## 状态存储与返回设计

### 1. 批量内存状态

当前 `batch_tasks` / `task_manager` 的批量状态需要扩展以下字段：

- `is_unlimited`
- `consecutive_failures`
- `max_consecutive_failures`
- `stop_reason`
- `domain_stats`

普通批量这些字段也可统一存在，只是：

- `is_unlimited = false`
- `consecutive_failures = 0`
- `domain_stats` 在结束后生成

### 2. WebSocket 状态推送

批量任务 WebSocket 状态消息同步追加上述字段。

这样前端在运行中即可实时更新：

- 当前是否无限模式
- 连续失败次数
- 是否进入自动停机收尾阶段

### 3. 结束时域名统计下发

任务结束时：

- REST `GET /registration/batch/{batch_id}` 返回 `domain_stats`
- WebSocket 最终状态消息也返回 `domain_stats`

前端优先使用 WebSocket 最终状态；若断线，则轮询状态接口拿到相同结果。

## 数据库设计

### 1. registration_tasks 扩展

建议为 `registration_tasks` 增加邮箱字段，例如：

- `email_address = Column(String(255), index=True, nullable=True)`

用途：

- 保存任务创建出的邮箱地址
- 支撑成功 / 失败统一统计
- 便于后续排查单个域名任务

### 2. 兼容性策略

历史任务没有该字段值时：

- 不影响新任务执行
- 仅在旧任务做统计时归类为 `未获取邮箱`

本轮不新增独立统计表，避免过度设计。

## 错误处理

### 1. 无限注册自动停止

当触发“连续失败超过 10 次”时：

- 写入批量日志说明停止原因
- 设置 `stop_reason = too_many_consecutive_failures`
- 状态进入“停止派发，等待收尾”
- 收尾完成后再置 `finished=true`

### 2. 手动取消

当用户手动取消时：

- 设置 `stop_reason = manual_cancel`
- 停止继续派发新任务
- 等待已运行任务自然完成
- 完成后输出最终统计

### 3. 普通批量完成

普通批量正常完成时：

- `stop_reason = completed`
- 生成域名统计
- 返回最终汇总

## 风险与控制

### 1. 无限模式资源占用风险

风险：无限注册可能长时间运行，占用较多浏览器会话、邮箱接口或代理资源。

控制方式：

- 保留并发上限控制
- 保留 interval 控制
- 通过连续失败阈值自动止损
- 继续支持手动取消

### 2. 并发下连续失败理解偏差

风险：若按创建顺序统计，会和用户看到的结果顺序不一致。

控制方式：

- 明确按完成顺序统计连续失败
- 在日志中记录连续失败计数变化

### 3. 域名统计数据缺失

风险：若邮箱地址只保存在成功结果里，失败任务统计会不完整。

控制方式：

- 在任务层单独保存邮箱字段
- 无邮箱时统一归类为 `未获取邮箱`

## 总结

本轮设计把注册数量扩展和无限注册模式视为同一个批量执行能力升级：

- 普通批量负责“已知数量”
- 无限注册负责“持续执行直到满足停止条件”

同时通过任务邮箱落点与结束态聚合，把批量任务结果从“只有总成功/失败数”提升为“可按域名分析成功率”的可观测状态。

在不推翻现有注册页面结构的前提下，这套方案能满足用户当前的核心诉求，并为后续继续扩展批量任务能力保留清晰边界。
