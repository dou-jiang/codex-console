# Pagination Jump and Filter Panel Design

**Date:** 2026-03-23

## Goal

为全站已分页或具备分页能力的列表补齐统一的“指定页跳转”体验，并将筛选区域升级为更专业的卡片式筛选面板。

## Confirmed Decisions

- 对“已有分页条”与“支持分页但尚未渲染分页条”的列表一并补齐分页 UI。
- 指定页跳转支持：
  - 输入页码后按 `Enter`
  - 点击“跳转”按钮
- 页码越界处理：
  - 自动修正到合法范围
  - 同时给出轻提示
- 筛选视觉方向：**卡片筛选面板风**

## In Scope

- 统一分页组件视觉与交互。
- 为账号管理列表增加“指定页跳转”。
- 为定时任务运行中心增加完整分页条与指定页跳转。
- 升级列表筛选区样式，使其成为统一的筛选面板。
- 提供最小必要的前端状态同步、页码修正和提示逻辑。

## Out of Scope

- 不引入复杂页码数字条（如 `1 2 3 4 ...`）。
- 不为当前没有后端分页能力的列表硬加伪分页。
- 不修改后端分页协议结构，除非现有响应完全缺少必要字段。
- 不重构业务数据加载方式。

## Current Problems

### 1. 分页能力不一致

- 账号管理页已有上一页/下一页，但无法直接跳页。
- 定时任务运行中心前端有 `page/pageSize` 过滤参数，但没有完整分页 UI，用户只能看到第一页结果。

### 2. 跳页效率低

当数据量较大时，用户只能连续点“下一页”，效率很差。

### 3. 筛选区域观感散乱

当前筛选控件大多直接堆在工具栏里，输入框、标签、按钮缺少统一层级和分组，显得不够专业。

## Proposed Approach

采用“**统一分页条 + 统一筛选面板**”方案：

1. 在全局样式中新增可复用的分页面板和筛选面板样式。
2. 在账号管理页保留现有分页逻辑，但补充：
   - 跳页输入框
   - 跳转按钮
   - Enter 支持
   - 自动修正与轻提示
3. 在定时任务运行中心正式渲染分页条，并基于已有 `scheduledRunFilters.page/pageSize` 驱动分页切换和跳页。
4. 将筛选区升级为卡片式条件面板，控件分组、按钮区和响应式换行统一处理。

## Pagination Design

### Shared UI shape

统一分页区域包含：

- 上一页按钮
- 当前页 / 总页数文本
- 指定页输入框
- 跳转按钮
- 下一页按钮

示例文案：

- `第 3 页 / 共 18 页`
- 输入框 placeholder：`页码`
- 按钮：`跳转`

### Shared behavior

- 当前无数据时：
  - 当前页显示为 `第 1 页 / 共 1 页`
  - 上一页 / 下一页 / 跳转均禁用
- 输入非数字：
  - 阻止跳转
  - 轻提示用户输入有效页码
- 输入小于 1：
  - 跳到 1
  - 轻提示“已自动跳到第 1 页”
- 输入大于总页数：
  - 跳到最后一页
  - 轻提示“已自动跳到最后一页”
- 若输入页码等于当前页：
  - 不重复请求
  - 可直接清空输入或保持原值

## Filter Panel Design

### Shared visual rules

- 筛选区使用独立卡片面板承载，不再与批量操作或普通按钮完全混排。
- 面板包含：
  - 标题/说明（可选）
  - 条件网格
  - 动作区（筛选 / 重置）
- 条件项统一标签字号、控件高度、间距和聚焦态。

### Layout rules

- 桌面端：多列网格，按字段重要性排列。
- 窄屏：自动折行为单列或双列，保证可点击面积和可读性。
- 按钮区固定在面板底部或右侧，避免条件控件与按钮混成一排。

## File-Level Design

- `static/css/style.css`
  - 增加：
    - `.filter-panel`
    - `.filter-panel-grid`
    - `.filter-panel-actions`
    - `.pagination-panel`
    - `.pagination-jump`
  - 统一输入框、说明文字、按钮组在筛选/分页场景下的视觉表现。

- `templates/accounts.html`
  - 给现有筛选区增加卡片面板语义 class。
  - 给现有分页区增加跳页输入框和跳转按钮。

- `static/js/accounts.js`
  - 新增页码输入 DOM 引用。
  - 封装跳页校验与修正逻辑。
  - 保持现有列表加载接口不变。

- `templates/scheduled_tasks.html`
  - 运行中心筛选区替换为筛选面板结构。
  - 新增运行中心分页条与跳页控件。

- `static/js/scheduled_tasks.js`
  - 保存并更新运行中心分页信息。
  - 支持上一页 / 下一页 / 指定页跳转。

## Data Expectations

### Accounts list

现有 `/accounts` 响应已包含：

- `accounts`
- `total`

前端可继续通过：

- `currentPage`
- `pageSize`
- `Math.ceil(total / pageSize)`

计算分页信息。

### Scheduled runs list

当前前端使用 `/scheduled-runs?...page=...&page_size=...` 请求，但只使用了 `items`。

设计要求前端优先兼容以下返回：

- `items`
- `total`
- `page`
- `page_size`

若后端当前未返回完整分页信息，则补最小必要字段，使运行中心可正确展示总页数与页码修正逻辑。

## Error Handling

- 跳页输入非法：toast warning，保持当前页不变。
- 跳页请求失败：恢复按钮可用态，并提示请求失败。
- 筛选后若总页数变小而当前页越界：
  - 自动重置到第 1 页重新加载，或使用服务端返回页码修正结果同步到前端。
  - 首选前端直接在变更筛选条件时把 `page` 重置为 `1`。

## Testing Strategy

- 账号管理页前端测试：
  - 跳页输入和按钮存在
  - Enter 跳转生效
  - 越界自动修正
- 定时任务页前端测试：
  - 分页条存在
  - 分页查询参数能跟随页码变化
  - 跳页和越界修正逻辑正确
- 模板/样式测试：
  - 筛选面板 class 存在
  - 分页 jump 相关 hook 存在

## Risks and Mitigations

### 风险 1：定时任务接口缺少总数

如果 `/scheduled-runs` 未返回总数，总页数无法精确计算。需要补充最小后端响应字段，保持与现有分页协议一致。

### 风险 2：不同页面分页逻辑重复

本次优先抽取“共享样式 + 轻量共享交互模式”，如有必要可在后续再抽更通用的 JS helper，避免一次性过度封装。

### 风险 3：筛选区变大影响纵向空间

通过可折行网格和紧凑控件尺寸控制面板高度，让其“更专业”但不过分占位。
