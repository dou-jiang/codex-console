# 代理管理增强与注册流程统一代理调度设计

## 背景

当前项目已经支持代理列表批量导入、基础筛选和批量删除，但仍存在三个明显问题：

1. 批量导入结果直接嵌在页面中，数据量一大就影响可读性
2. 代理列表仍然不适合更大规模数据管理，需要服务端分页
3. 代理检测只能“全部测试”，不能指定目标地址，也不能只测试勾选项
4. 更关键的是，注册流程虽然有动态代理、全局静态代理和代理列表三套概念，但并没有统一调度层，导致“代理已导入，但注册流程未实际使用代理列表”的问题

本轮设计目标是在不重做整站结构的前提下，同时增强代理管理页交互，并为注册流程引入统一代理调度层。

## 目标

- 将批量导入结果改为弹窗展示，并在内容过长时支持滚动
- 将代理列表改为服务端分页
- 新增代理检测弹窗，支持：
  - 自定义目标检测地址
  - 检测当前勾选代理
  - 检测全部代理
- 为注册流程新增统一代理调度层
- 确保注册流程真正消费代理列表
- 落实代理优先级和失败重试规则

## 非目标

- 不在本轮把登录、支付、刷新 token 等流程全部接入统一代理调度层
- 不新增复杂的代理熔断、统计分析、历史报表页面
- 不把代理检测扩展成长期任务系统
- 不重构整个设置页面为独立 SPA

## 用户确认的关键规则

- 代理列表分页方式：服务端分页
- 代理检测弹窗检测范围：弹窗里二选一（检测当前勾选 / 检测全部）
- 需要排查并修复“注册流程没有使用导入代理”的问题
- 注册流程的代理列表选择规则：默认代理优先，没有默认时从启用代理中随机选一个
- 注册流程代理来源优先级：动态代理 > 代理列表 > 全局静态代理
- 代理失败时：自动换下一个启用代理重试
- 本轮最终实现方向：方案 3（统一代理调度层）

## 总体方案

本轮按两个子系统推进：

1. **代理管理页增强**
   - 导入结果弹窗
   - 服务端分页
   - 检测弹窗

2. **注册流程统一代理调度层**
   - 把动态代理、代理列表、全局静态代理纳入一个统一候选链
   - 由注册流程通过调度层获取当前代理和下一个代理
   - 代理相关失败时自动切换

这两部分共享同一个代理数据源，但职责分离：

- 管理页负责“管理代理池”
- 调度层负责“消费代理池”

## 管理页交互设计

### 1. 批量导入结果弹窗

将当前页面内嵌的导入结果区域改成弹窗：

- 导入完成后自动弹出
- 顶部展示：
  - 成功数
  - 跳过数
  - 失败数
- 中部展示逐行结果
- 结果区固定高度，内容超长时显示纵向滚动条
- 支持关闭弹窗后继续停留在代理列表页

结果数据仍沿用现有导入返回结构，但前端展示改为 modal。

### 2. 代理列表服务端分页

代理列表从“一次返回全部”改为服务端分页：

- 支持 `page`
- 支持 `page_size`
- 前端展示：
  - 当前页码
  - 总页数
  - 上一页 / 下一页
  - 每页条数选择（建议 20 / 50 / 100）

分页与现有筛选联动：

- 修改筛选条件时自动回到第一页
- 清空筛选时恢复第一页的默认列表

### 3. 代理检测弹窗

新增检测弹窗，替代单纯“测试全部”按钮：

弹窗内容包含：

- 目标检测地址输入框
- 检测范围单选：
  - 当前勾选代理
  - 全部代理
- 开始检测按钮

检测结果同样在弹窗中展示：

- 总数 / 成功 / 失败
- 每个代理的检测结果明细
- 失败原因 / 响应耗时 / 出口 IP（如果能拿到）

结果内容超长时启用滚动条。

### 4. 列表多选联动

代理列表保留多选机制，用于：

- 批量删除
- 检测当前勾选代理

当未勾选任何代理时，检测弹窗仍可打开，但：

- 若用户选择“检测当前勾选”，前端给出明确提示
- 若用户选择“检测全部”，则继续正常执行

## 注册流程统一代理调度层设计

### 1. 调度层职责

新增一个内部代理调度模块，专门用于注册流程代理选择。

职责包括：

- 构建本次注册任务的候选代理序列
- 向注册流程提供当前可用代理
- 当代理失败时切换到下一个候选代理
- 记录最终成功代理或失败尝试链路

调度层本身不负责发请求、不负责页面展示，只负责“选代理和切代理”。

### 2. 候选代理优先级

按用户确认的顺序执行：

**动态代理 > 代理列表 > 全局静态代理**

具体规则：

1. 若启用动态代理并成功获取代理
   - 作为最高优先级候选

2. 若动态代理不可用或失败
   - 进入代理列表候选池
   - 候选规则：
     - 默认代理优先
     - 其它启用代理作为后备
     - 其它启用代理随机打散，避免长期集中命中同一批代理

3. 若代理列表没有可用代理
   - 回退到全局静态代理设置

### 3. 注册流程接入方式

注册任务启动时：

1. 调度层生成候选代理序列
2. 注册执行层取第一个代理进行请求
3. 如果识别为代理相关失败，则切换下一个候选代理
4. 直到：
   - 某个代理成功完成关键注册步骤
   - 或候选序列耗尽

### 4. 失败分类规则

只有“代理相关失败”才触发代理切换，例如：

- 连接失败
- 超时
- 认证失败
- 代理出口不可用
- 明显的代理链路错误

以下情况不自动换代理：

- 邮箱验证码失败
- OpenAI 页面结构变化
- 非代理维度的业务失败
- 参数错误或代码错误

也就是说，注册流程负责识别“这是不是代理问题”，调度层负责“如果是，就切下一个”。

### 5. 成功与尝试记录

为便于排查，建议记录：

- 本次候选代理序列
- 实际尝试过的代理列表
- 最终成功使用的代理
- 若失败，失败时卡在哪个代理上

这些信息可以优先记录到任务日志或任务结果 JSON 中，不强制要求先扩表。

## 后端接口设计

### 1. 代理列表分页接口

扩展现有：

- `GET /settings/proxies`

新增查询参数：

- `page`
- `page_size`
- `keyword`
- `type`
- `enabled`
- `is_default`
- `location`

返回建议结构：

- `items`
- `total`
- `page`
- `page_size`
- `total_pages`

### 2. 批量导入接口

沿用现有批量导入接口：

- `POST /settings/proxies/batch-import`

返回结构保留：

- `success`
- `skipped`
- `failed`
- `results`

建议在返回前按 `line_no` 排序，便于弹窗按原始输入顺序展示。

### 3. 批量检测接口

新增：

- `POST /settings/proxies/test-batch`

请求体：

- `mode: "selected" | "all"`
- `ids: number[]`（仅当 `mode=selected` 时使用）
- `target_url: str`

返回：

- `total`
- `success`
- `failed`
- `results[]`

每条结果建议包含：

- 代理 ID
- 名称
- success / failure
- message
- response_time
- exit_ip（可选）

## 调度层内部接口设计

调度层不暴露 HTTP API，优先作为注册流程内部服务。

建议提供以下接口：

### 1. `build_proxy_candidates_for_registration()`

职责：

- 根据当前设置和数据库状态生成候选代理序列

### 2. `get_next_proxy()`

职责：

- 返回当前候选代理
- 或在失败后返回下一个候选代理

### 3. `report_proxy_failure(...)`

职责：

- 标记当前代理失败
- 推进到下一个代理

### 4. `report_proxy_success(...)`

职责：

- 标记当前代理成功
- 记录最终使用代理

## 数据结构设计

### 代理表

现有 `Proxy` 模型继续沿用：

- `enabled`
- `is_default`
- `last_used`
- `country`
- `city`

无需在本轮大改主表结构。

### 任务记录

建议在注册任务结果或日志中记录：

- `proxy_candidates`
- `proxy_attempts`
- `proxy_used`

优先放入 result JSON / logs，而不是立即新增数据库字段。

## 实施顺序建议

为降低风险，建议按下面顺序实现：

1. 先实现调度层 + 单元测试
2. 再把调度层接入注册流程 + 集成测试
3. 再补分页接口与批量检测接口
4. 最后做管理页 UI 改造（弹窗 / 分页 / 检测弹窗）

这样能避免“一边改 UI，一边改注册核心链路”导致问题难定位。

## 风险与注意事项

### 1. 注册流程侵入风险

调度层接入注册流程时，需要特别注意不要破坏当前注册链路稳定性。

### 2. 阻塞风险

代理检测和代理地理位置查询都可能涉及阻塞 I/O，需要避免直接阻塞 async 路由事件循环。

### 3. 一致性风险

分页、筛选、批量选择、批量检测、批量删除要共享一致的列表查询语义，避免“UI 看到的是一套，检测/删除用的是另一套”。

## 成功标准

- 导入结果改为弹窗展示，超长时可滚动
- 代理列表支持服务端分页
- 代理检测支持弹窗、自定义目标地址、检测勾选或全部
- 注册流程通过统一代理调度层使用代理
- 注册流程满足：动态代理 > 代理列表 > 全局静态代理
- 代理列表满足：默认代理优先，其它启用代理后备
- 代理失败时能自动切换下一个启用代理
- 现有单代理管理功能不回退
